## 0. Executive Summary & Headline Finding

**Power-up is mostly pre-built infrastructure, but it requires three genuinely new pieces of infrastructure (not reuse): a new `RestrictionExpiry` temporal anchor for Kang (B1), a parameterization of `ProhibitedActivity::ActivateAbilities` to scope the prohibition to a tag (B2), and threading the activating ability's tag through the cost-payment chain into `PaymentContext::Activation` for Quinjet (M4/M5).** The engine already has an `AbilityTag`-based labeled-activated-ability subsystem (Boast/Outlast/Cycling/Exhaust/Evolve/Backup) with per-tag activation-limit statics, per-tag activation-cost-reduction statics, per-tag/per-source per-game and per-turn activation counters, a tag-carrying activation event, a tag-keyed trigger, and a self-referential `CostReduction` struct. Combined with the **already-existing** `QuantityRef::SelfManaValue` and `ParsedCondition::SourceEnteredThisTurn`, the dynamic "reduce the cost by his mana cost if he entered this turn" reduction is expressible **with zero new quantity/condition types**.

The plan adds:
1. **One new `AbilityTag::PowerUp` variant** (leaf; correct).
2. **One new `ManaRestriction::OnlyForTaggedActivation(AbilityTag)` variant** + threading the activating ability's tag through the cost-payment chain to the spend gate (M4/M5).
3. **A parameterization of `ProhibitedActivity::ActivateAbilities`** with `only_tag: Option<AbilityTag>` (B2).
4. **A new `RestrictionExpiry::UntilEndOfNextTurnOf { player }` variant** (B1) ‚Äî pre-armed at lowering, converted to the existing `RestrictionExpiry::EndOfTurn` at the target player's untap step (mirroring the existing `Duration::UntilEndOfNextTurnOf` arming idiom in `layers.rs::prune_until_next_turn_effects`); the existing cleanup-step prune then ends it. **No new cleanup-prune code, no `bool` latch field.**
5. **A per-game limit path** for Wonder Man (M1), sharing only the static-scan with the existing per-turn path.
6. **Parser dispatch** for the `Power-up ‚Äî` line + four static/restriction/trigger phrases (all nom).

Honestly deferred: **Advancing the Spirit** (per-turn alternative-cost replacement keyed on power-up abilities) ‚Äî out of scope; remains an unsupported coverage gap.

## 2. Mandatory Architectural Sections

### 2.1 Pattern Coverage
Power-up is a **keyword-labeled activated-ability class** ‚Äî the same architectural shape as Boast/Outlast/Cycling/Exhaust. The base-case primitive (tagged activated ability with per-game limit + self-MV cost reduction) **directly covers 37 cards**. The six named cards exercise independent reusable axes (class-scoped static cost reduction, class-scoped activation-limit static, tag-scoped mana restriction, tag-scoped activation prohibition, tag-scoped trigger), each generalizing to **any future labeled-ability mechanic** and hardening the shared `AbilityTag` subsystem. No axis is a one-card special case. **Distinct surfaces: 37 base-case + 6 named (Hulk/Wonder Man/Kang each also appear in the base bucket, so the union of named+base is ~37 cards); 1 deferred outlier (Advancing the Spirit).** The prior plan's claim that this "unlocks the entire power-up cycle" is **removed** ‚Äî Advancing the Spirit's per-turn alt-cost replacement is a real, honest gap.

### 2.8 Variant Discoverability ‚Äî `/add-engine-variant` gate results
`grep data/engine-inventory.json` confirmed: **no `PowerUp` / `power-up` references** (clean slate). Four variant proposals routed through the gate:

**(a) `AbilityTag::PowerUp` ‚Äî APPROVED (new sibling leaf, correct).**
- *Parameterize-vs-sibling*: `AbilityTag` is a flat enumeration of keyword origins (Boast/Evolve/Exhaust/Outlast/Cycling/Backup). Each tag is an opaque identity class label with no comparator/scope/aggregator axis to lift. Power-up is a new keyword origin ‚Üí a new leaf is the only correct shape. **Not** a sibling-cluster smell.
- *Categorical boundary*: all `AbilityTag` variants live in keyword-ability rule space (CR 702.x) + activation-event machinery (CR 602.1). Single section. ‚úî
- *Lockstep*: forces a new arm in the wildcard-free `match` inside `effective_activation_limit` (`restrictions.rs:577-589`) ‚Üí promoted to `AbilityTag::keyword_str()`. The emit/matcher paths are tag-agnostic (no change). ‚úî

**(b) `ManaRestriction::OnlyForTaggedActivation(AbilityTag)` ‚Äî APPROVED (new sibling along a genuinely new axis).**
- *Parameterize-vs-sibling*: `OnlyForTypeSpellsOrAbilities { spell_type, ability: AbilityActivationScope }` keys on the **source permanent's type** (`allows_activation` is `fn(&self, source_types, source_subtypes)` ‚Äî `mana.rs:471`); `OnlyForActivation` permits **any** activation. Neither expresses "only power-up-tagged abilities" ‚Äî the discriminating axis is the **ability's keyword tag**, not the permanent's type. New structural axis ‚Üí new variant. (A parameterization of `OnlyForTypeSpellsOrAbilities` would conflate permanent-type with ability-tag, two different concerns.) Generalizes to any future tag. Built for the class.
- *Categorical boundary*: CR 106.6 (mana spend restriction) ‚Äî same section as every other `ManaRestriction`. ‚úî
- *Lockstep*: requires (1) extending `PaymentContext::Activation` with `ability_tag: Option<AbilityTag>`, (2) changing `allows_activation`'s signature to receive the tag and adding the arm, (3) threading the activating tag through the cost-payment **signature family** to all production `PaymentContext::Activation` construction sites (M4 ‚Äî see reframed scope below), (4) lowering `ManaSpendRestriction::ActivateTagged ‚Üí ManaRestriction::OnlyForTaggedActivation` in `effects/mana.rs`. Exhaustive matches + the new ctx field enforce completeness. ‚úî
- *Reuse consideration*: evaluated reusing `OnlyForAny` (disjunctive combinator) ‚Äî rejected; the inner branch still needs a leaf that matches an ability tag, which does not exist. The leaf is the new variant; `OnlyForAny` remains the composition layer above it.

**(c) `ProhibitedActivity::ActivateAbilities { exemption, only_tag: Option<AbilityTag> }` ‚Äî APPROVED (parameterization, preserves existing axis).**
- *Parameterize-vs-sibling*: the existing variant `ActivateAbilities { exemption: ActivationExemption }` prohibits **all** ability activations for affected players (optionally exempting mana abilities). Kang prohibits **only power-up** abilities. This is a *restricting* parameterization of the same variant along a new optional axis ‚Äî NOT a new sibling. ## 4. Genuine Human Design Decisions to Resolve

1. **Wonder Man "additional time" scope (per-game vs per-turn).** Power-up's base limit is once-per-**game**. "Can be activated an additional time" with no qualifier is read as **per game** (cap 2 total). *Recommendation: per-game.* Low risk; worth a maintainer nod (if an official ruling says per-turn, the modeling would attach to a per-turn cap instead).
2. **`RestrictionExpiry::UntilEndOfNextTurnOf` arming idiom vs turn-number stamp.** The plan mirrors the existing `Duration::UntilEndOfNextTurnOf` convert-at-untap idiom (convert to `EndOfTurn` at the target turn's untap, pruned at its cleanup), with NO `bool` field. An alternative is stamping the target `turn_number` at lowering time. *Recommendation: the convert-at-untap mirror* ‚Äî it reuses the codebase's existing, reviewed pattern and its cleanup prune; a turn-number stamp would mis-fire under extra-turn LIFO insertion (other extra turns shifting the number). Confirm preference.
3. **`UntilEndOfNextTurnOf` lowering split.** Splitting `Duration::UntilEndOfNextTurnOf { Controller }` away from `UntilPlayerNextTurn` (to the new mirror anchor) for restrictions is the correct fix for Kang. Confirm no other card relies on the old (start-of-next-turn) lowering for a *restriction* (the existing doc comment claims none does ‚Äî Kang is the first).
4. **Coarse `CantActivateAbilities` player-status projection (B2).** `derived_views.rs:392` projects a power-up-only prohibition as the generic "can't activate abilities" status. Correctness is unaffected (the consult site is precise); only the displayed player-condition is coarse. *Recommendation: leave coarse for this wave* unless a finer status is desired.
5. **Skipped/prevented-extra-turn arming-leak edge (FIX 3).** If Kang's granted extra turn is skipped or prevented (a `turns_to_skip` decrement or a turn-start replacement `Prevented`) landing exactly on that turn, the untap-step conversion never runs and the `UntilEndOfNextTurnOf` restriction never expires (wrongly blocking later power-up abilities). The same edge exists in the analogous `Duration::UntilEndOfNextTurnOf` arming. *Recommendation: accept as a documented narrow edge for this wave* (comment at the conversion site + a recorded known-leak test anchor), pending a future shared robust-arming fix that would cover both `Duration` and `RestrictionExpiry`. Confirm acceptance.

All other axes are mechanical reuse of verified existing infrastructure.

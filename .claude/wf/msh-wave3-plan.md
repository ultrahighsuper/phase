## Analogous trace (end-to-end)
Pipeline: effect/combat proposes `ProposedEvent` → `replacement::replace_event()` → `pipeline_loop()`
→ `find_applicable_replacements()` (gating) → registry matcher/applier → `ApplyResult::{Modified,Prevented}`.
CR 614.5 idempotence: `pipeline_loop` calls `proposed.mark_applied(rid)` BEFORE the applier and threads
`applied: HashSet<ReplacementId>` through the Modified event.

- CreateToken: `effects/token.rs::resolve()` → `ProposedEvent::CreateToken{owner,spec:Box<TokenSpec>,count,…}`
  → `create_token_matcher` (replacement.rs:2202) → `create_token_applier` (2206) reads quantity_modification /
  additional_token_spec / ensure_token_specs / token_owner_redirect. NO field replaces the spec wholesale.
- DamageDone (source): `effects/deal_damage.rs::apply_damage_to_target` → `ProposedEvent::Damage{source_id,target,amount,is_combat,applied}`
  → `damage_done_matcher`(849) → `damage_done_applier`(911) applies `DamageModification::{Double,Triple,Plus,Minus,SetTo,SetToSourcePower,LifeFloor}`.
  **`Double` already exists (Furnace of Rath).** `damage_source_filter` evaluated in `find_applicable_replacements`
  (replacement.rs:3996) via `matches_target_filter(source_id, sf, FilterContext::from_source_with_controller(...))`
  — `TargetFilter::AttachedTo` resolves to "the object the equipment is attached to". (See the Mjölnir
  classification guard below — the existing :8642 test exercises AttachedTo via a DIFFERENT path; the
  `matches_target_filter` source-filter path must be confirmed at impl time.)
- DealtDamage (receiver): SAME `ProposedEvent::Damage` variant → `dealt_damage_matcher`(2758, fires when
  `target == replacement source object`) → `dealt_damage_applier`(2774) is **currently a no-op** (`Modified(event)`).
  Marked damage = `GameObject.damage_marked: u32`; umbra armor & regeneration clear it inline
  (`obj.damage_marked = 0; obj.dealt_deathtouch_damage = false;` at replacement.rs:1518-1522).
  No standalone heal Effect exists (`Effect::Regenerate` also taps/removes-from-combat — wrong).

## Combat-damage delivery model (REQUIRED reading for Wolverine — traced this revision)
`apply_combat_damage` (combat_damage.rs:814) is strictly three-phased (doc comment :800-810):
- **Phase A (Collect, :824-867):** build every `ProposedEvent::Damage` via `pre_replacement_damage_gate`.
  NO damage marked.
- **Phase B (Batch replace, :869-871):** `replace_combat_damage_batch` (replacement.rs:5640) loops
  `pipeline_loop` PER-EVENT (:5651-5682). This is where EVERY DealtDamage applier (incl. Wolverine's)
  fires. **No event marks `damage_marked` here** — appliers only return `ApplyResult::Modified(event)`.
- **Phase C (Apply, :873-944):** for each survivor, `apply_damage_after_replacement` (deal_damage.rs:358)
  runs. The **sole** `damage_marked += actual_amount` increment is deal_damage.rs:446, reached only from
  `apply_damage_after_replacement`. Phase B never calls it.

**Consequence (the simultaneity guarantee):** because all Phase-B appliers complete before ANY Phase-C
delivery, at the instant Wolverine's heal runs in Phase B, `damage_marked` holds exactly the PRE-BATCH
value (prior turn/step damage), and ZERO same-batch combat instances are marked yet. The phase boundary
IS the pre-batch snapshot. CR 510.2 (all combat damage dealt simultaneously) is satisfied structurally.
The noncombat path (`apply_damage_to_target`, deal_damage.rs:292→307) is a single `replace_event` then a
single `apply_damage_after_replacement` — same heal-before-mark ordering, no batch, no same-batch concern.

source-filter path Mjölnir actually uses. These are different resolvers. **Implementer instruction (do this

equipped creature deals combat damage → assert DOUBLED) against the CURRENT engine with a hand-built

- CREATURE-TOKEN GATE: new `ReplacementCondition::TokenCoreTypeMatches { core_types: Vec<CoreType> }` (parallels

### Card 3 — Wolverine, Fierce Fighter (receiver heal; HIGHEST effort; LAST)
"If damage would be dealt to Wolverine, instead that damage is dealt, but all other damage already dealt to him is healed."
Gaps: (a) `parse_damage_to_self_instead_followup` (:6155) matches "if damage would be dealt to ~, <effect> instead"
— Wolverine's "instead" is mid-clause and "him" is not normalized to ~. (b) `dealt_damage_applier` is a no-op.
(c) the heal must run PRE-delivery so only PRIOR (pre-batch) marked damage clears while the new instance lands.

**Wolverine simultaneity resolution — OPTION (i) CHOSEN (no deferral).** The heal clears only damage marked
BEFORE the current combat batch; same-batch simultaneous instances are preserved (CR 510.2). This wave ships it
live and combat-batch-correct.

**Tractability evidence (why option (i) is single-seam, not multi-day, and why NO explicit snapshot map is added):**
Per the "Combat-damage delivery model" trace above, `replace_combat_damage_batch` runs every DealtDamage applier in
**Phase B**, and the *only* `damage_marked` increment (deal_damage.rs:446, inside `apply_damage_after_replacement`)
runs in **Phase C**, after the batch returns. Therefore a heal placed in `dealt_damage_applier` (Phase B) that sets
`damage_marked = 0` clears exactly the pre-batch value — at that instant zero same-batch combat instances are marked —
and Phase C then marks each survivor (3+2 = full simultaneous total). The reviewer-requested "pre-batch marked-damage
snapshot" is therefore **already provided structurally by the Phase-B/Phase-C boundary**; threading an explicit
per-object snapshot map (mirroring `combat_prevention_tally`) would duplicate a guarantee the architecture already
enforces and is the kind of redundant validation CLAUDE.md forbids. We commit to option (i)'s correctness contract
("only pre-batch damage heals; same-batch preserved") with the minimal correct mechanism, and PROVE it with the
discriminating gang-block test below (which revert-fails the instant the heal is moved to delivery time / interleaved).
The load-bearing invariant (Phase B precedes Phase C; deal_damage.rs:446 is the sole marking site) is documented in a
code comment at the applier so a future interleaving refactor is caught by both the comment and the red test.

Approach (A) implementation:
- New parser arm (dispatch beside self/player followups :280-283): recognize "if damage would be dealt to
  {~|him|her|it|them}, instead that damage is dealt, but all other damage already dealt to {~|him|…} is healed"
  via composed `tag()`/`alt()` (object-pronoun alt for receiver self-ref). Emits a DealtDamage replacement with NO
  prevention_shield, carrying a typed heal instruction.
- New reusable `Effect::RemoveAllDamage { target: TargetFilter }` primitive (genuine MTG "remove all damage from
  target creature"), carried in `execute`. `dealt_damage_applier` recognizes it and applies inline PRE-delivery
  (Phase B): set receiver `damage_marked = 0; dealt_deathtouch_damage = false` (mirrors umbra/regen
  replacement.rs:1518-1522) BEFORE returning `Modified(event)` unchanged. Add a comment at this site documenting
  the Phase-B-before-Phase-C invariant (combat_damage.rs:869-944 / deal_damage.rs:446) that makes the heal
  combat-batch-correct. Standard `/add-engine-effect` (resolver in effects/mod.rs, EffectKind, targeting, AI/FE
  serialization, coverage).
- **mtgish-safety (CITATION FIX #1, CONFIRMED):** adding `Effect::RemoveAllDamage` cannot break mtgish. mtgish-import
  only ever CONSTRUCTS engine `Effect` / `ReplacementCondition` values — it never exhaustively matches the engine
  `Effect` enum. Where it does match over `Effect`, every site has a catch-all: `rewrite_bound_x_in_effect`
  (convert/action.rs:451), `rewrite_any_target_filter_in_effect` (:597), `apply_player_target` (:5458, a named
  `other =>` strict-fail arm), and `apply_token_flags` (:6459) — all fall through on unrecognized variants. (The
  previously-cited `convert/token.rs:607` was WRONG: that line is a `match effect { TokenCopyEffect::AddAbility… }`
  over `TokenCopyEffect`, an mtgish AST type, NOT engine `Effect` — removed.) Conclusion unchanged: mtgish-safe.
  Re-confirm these catch-alls compile during impl (engine-only build), but no mtgish edits are needed or permitted.
CR: 614.1a, 510.2 (combat damage simultaneous — same-batch instances must not heal each other),
120.6 (marked damage remains until cleanup — heal removes early), 120.1/120.3, 614.5.
